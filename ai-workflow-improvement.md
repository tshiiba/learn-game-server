# AI連携によるワークフロー改善まとめ

## 1. 現状の業務フロー

ソーシャルゲームのサーバーエンジニア（リーダー）としての業務フロー：

1. 他社作成のゲーム仕様書（パワーポイント）を読み、BacklogにAPI単位でチケットを作成
2. チケットを他のサーバーエンジニアに割り振り（自分が担当する場合もあり）
3. チケット上でデータ設計（マスタデータ・ユーザーデータ）、API設計（リクエスト/レスポンス）を行う
4. 実装（AI活用 or 自分で書く）
5. AIに単体テストコードを書かせる
6. ローカルでサーバー起動、動作確認
7. コミット → Push → PR作成
8. GitHub Copilot Reviewへレビュー依頼（Backlogチケット紐づけは手動）
9. レビュー修正
10. サーバーリードエンジニアによるレビュー → 承認 → developへマージ

### 現状のAI活用状況

- 実装の一部をAIに任せている
- 単体テストコードはAIが生成
- AIへの指示は**口頭で説明**
- AIの出力が既存アーキテクチャに従わないことがある（新しいディレクトリを作成される等）
- 大幅修正は少ないが、想定と違う実装になることがある

## 2. 改善施策一覧

### 優先度：高

#### 2-1. `.github/copilot-instructions.md` の作成

プロジェクトのアーキテクチャ規約・ディレクトリ構造・命名規則を記述するファイル。
Copilotが**毎回自動的に読み込む**ため、「既存アーキテクチャに従わない」問題を直接解決する。

記述内容の例：
- ディレクトリ構造と各層の責務（handler / usecase / domain / repository / infrastructure）
- 新しいディレクトリを作成しないルール
- 命名規則
- ビジネスルールはdomain層に純粋関数として配置するルール

#### 2-2. Prompt Files の作成（`.github/prompts/*.prompt.md`）

繰り返し使う指示をテンプレート化し、毎回の口頭説明を不要にする。

##### `api-design.prompt.md`（設計フェーズ用）

画面ベースの仕様書からAPI設計を導出するためのフレームワーク：

1. **画面分析**（3つの視点）
   - **表示**: 画面描画に必要なデータは何か？
   - **操作**: ユーザーが実行できる操作は何か？
   - **裏側の駆動**: 画面に見えないが必要な処理は何か？
2. **API候補の列挙**: API名 / 種別(参照/更新) / トリガー条件
3. **各APIの設計**: リクエスト/レスポンス/バリデーション/ビジネスルールチェック/異常系
4. **ドメインルールの抽出**: 「〜は◯個」「〜は1日1回」等のルールを整理

##### `api-implement.prompt.md`（実装フェーズ用）

プロジェクト構成ルールと実装方針：
- 各層の配置ルール
- ビジネスルールはdomain層に純粋関数として実装
- usecase層はドメインルールを呼び出しリポジトリで永続化
- handler層はリクエスト変換とレスポンス組み立てのみ

##### `unit-test.prompt.md`（テスト生成用）

テスト方針：
- domain層: 純粋関数テスト（DB不要）
- usecase層: リポジトリをモックしてテスト
- 境界値・異常系を必ず含める

### 優先度：中

#### 2-3. Backlog MCP サーバーの導入

Copilot ChatからBacklogを直接操作可能にする。

**できること：**
- 仕様書からチケットを直接作成（ブラウザ不要）
- AIがチケット内容を自分で読みに行く（口頭説明不要）
- 設計結果をチケットにコメント投稿
- チケットのステータス変更

**最大の効果：** AIがチケットを直接読めるため、口頭説明のコストと情報欠落がなくなる

**活用イメージ：**

```
「添付した仕様書からAPI単位でBacklogにチケットを作成して。プロジェクトキーはAIEXP」
→ Copilotが直接Backlogにチケット作成

「AIEXP-101の内容を読んで、protoファイルとDDLを作成して」
→ CopilotがBacklogからチケット取得 → 実装生成
```

#### 2-4. 仕様書のAIドラフト化

パワーポイントの仕様書をスクリーンショット（画像化）してCopilot Chatに添付。
- **PowerPoint(.pptx)は直接読み込み不可**
- 画像（PNG/JPEG）としてドラッグ＆ドロップで添付するのが最も効果的
- テキストだけコピーすると図やレイアウト情報が失われるため、画面ベース仕様書には画像添付が適切

#### 2-5. `.github/copilot-review-instructions.md` の作成

Copilot Reviewにプロジェクト固有の観点でレビューさせる。

記述内容の例：
- マスタデータ参照にキャッシュを使っているか
- トランザクション境界は適切か
- プロジェクト固有のコーディング規約に準拠しているか

#### 2-6. 設計（proto/DDL）のAIドラフト生成

既存のprotoファイルやDDLを参照させることで、命名規則やスタイルを踏襲した設計ドラフトを生成。

### 優先度：低

#### 2-7. PR作成・Backlog紐づけ自動化

- ブランチ名にチケットID（例: `feature/AIEXP-123-add-gacha-api`）
- PRテンプレートやGitHub Actionsで自動リンク
- PR説明文のCopilot自動生成

## 3. 動作確認の効率化

### 推奨アプローチ

| 方法 | 効果 | おすすめ度 |
|------|------|-----------|
| Agent モードでターミナル操作を任せる | 手動の動作確認をAIに代行 | ○ |
| DB MCP サーバー導入 | DB状態の確認がChat内で完結 | ○ |
| **インテグレーションテストをAIに書かせる** | **手動確認自体が不要になる** | **◎** |

### 理想のフロー

```
現在:  実装 → 手動でサーバー起動 → 手動でリクエスト → 目視確認
理想:  実装 → AIがテスト生成 → go test で自動検証
```

MCP で動作確認を自動化するよりも、テストコードの自動生成で動作確認そのものを不要にする方が長期的なリターンが大きい。

## 4. クリーンアーキテクチャでの実装例（デイリーミッション）

### ディレクトリ配置

```
internal/
  domain/mission/
    mission.go          ← エンティティ・値オブジェクト
    mission_rule.go     ← 「1日3つ発生」などのドメインルール
  usecase/mission/
    generate_daily.go   ← デイリーミッション生成ユースケース
    get_daily.go        ← ミッション一覧取得
    receive_reward.go   ← 報酬受取
  repository/mission/
    repository.go       ← MissionRepository インターフェース
  infrastructure/mission/
    repository_impl.go  ← MissionRepository の実装
  handler/mission/
    handler.go          ← gRPCハンドラ
```

### ポイント

- 「1日3つ」というルールは **domain層** に純粋関数として配置
- usecase層はドメインルールを呼び出すだけ
- domain層のテストはDB不要（純粋関数テスト）
- usecase層のテストはリポジトリをモック
